# 交通决策治理智能体（第一阶段）设计稿

日期：2026-04-19  
状态：已评审（待实现计划）  
范围：产品形态与展示闭环优先，算法复杂度次要

## 1. 背景与目标

当前项目已具备聊天主链路与工具调用能力，但仍存在以下问题：

1. 首页和聊天文案仍带有“法律问答”定位残留。
2. 聊天页默认会自动进入历史会话，演示时会出现旧问题。
3. 地图结果主要依赖第三方原生 H5 嵌入，存在加载失败、滑块验证、扫码弹层等问题。
4. 缺少统一的“工具调用结果页”机制，后续模块难复用。

第一阶段目标：

1. 平台定位切换为“交通决策治理智能体”。
2. 聊天主界面保持核心入口，打通“问答 -> 工具调用 -> iframe结果页”的可演示闭环。
3. 工具结果统一为“文本总结 + 工具标签 + iframe 页面”。
4. 优先落地“城市拥堵体检问答”，并可复用到路线/周边模块。

## 2. 关键约束与确认

1. 不新增复杂算法，不引入重型新模块。
2. 第一阶段允许使用模板/规则/假数据。
3. 报告为会话临时生成，不做长期报表库。
4. 当前会话历史可回看报告。
5. 统一结果页路由：`/embed/report`。
6. 不需要天气模块（从主交互链路下线）。
7. 地图画布方案：在**自有 iframe 页面中嵌入第三方 JS 地图 SDK**，不再 iframe 第三方原生 H5 页面。
8. 第三方地图优先：高德 JS API 2.0（与现有后端高德 WebService 数据源一致）。

## 3. 总体方案

### 3.1 聊天链路不变，扩展统一展示协议

SSE 协议仍为 `type=content|done|sources`，不改接口路径。  
在 `content` 文本中输出可渲染的统一展示片段：

1. 自然语言总结。
2. 工具调用标签（已调用工具：xxx）。
3. iframe 容器（src 指向 `/embed/report?...`）。

### 3.2 会话内可回看的报告载体

采用 URL 携带轻量结构化数据：

- `iframe_url=/embed/report?type=<report_type>&data=<base64url(json)>`

说明：

1. 不新增 `report` 数据库表。
2. 报告元数据通过 AI 消息 content 一起落库，历史会话自然可回看。
3. payload 控制为摘要级，避免 URL 过长。

### 3.3 工具统一返回结构（后端内部约定）

工具函数返回 JSON 字符串，最小字段：

1. `text_data`: 文本摘要
2. `tool_name`: 工具名
3. `display_type`: 固定 `iframe_report`
4. `iframe_url`: 自有结果页地址

示例：

```json
{
  "text_data": "已生成主城区拥堵体检摘要，详见报告页。",
  "tool_name": "congestion_check",
  "display_type": "iframe_report",
  "iframe_url": "/embed/report?type=congestion&data=..."
}
```

## 4. 模块设计

### 4.1 首页（Landing）

保留现有卡片布局，文案和入口统一为交通治理方向：

1. 交通治理助手
2. 城市拥堵体检
3. 仿真推演评估
4. 个人中心

前三个入口进入 `/chat`，个人中心进入 `/profile`。  
移除法律属性入口词汇与管理员额外入口展示。

### 4.2 聊天页（Chat）

1. 进入 `/chat` 时默认新建空白会话，不自动切历史。
2. 欢迎区与推荐问题改为交通治理方向。
3. 输入框占位改为交通运行/拥堵体检/治理决策语义。
4. 支持在 AI 消息中显示：
   - 工具标签
   - 统一 iframe 容器
5. 保留旧历史 `iframe` 过滤兜底，防止旧消息触发异常嵌入。

### 4.3 工具与 Agent

第一阶段保留并统一三类工具：

1. `congestion_check`（新增，核心）
2. `route_plan`（改造）
3. `nearby_search`（改造）

处理规则：

1. 命中工具后先输出状态文本（例如“正在生成拥堵体检报告...”）。
2. 工具返回统一结构后，输出文本总结 + 工具标签 + iframe HTML。
3. 完成后及时输出 `done`，防止前端长期 loading。

天气工具策略：

1. 从 `Agent` 默认工具列表移除。
2. 不在欢迎推荐中出现。
3. 旧实现代码可先保留，避免额外回归风险。

### 4.4 统一报告页（/embed/report）

新增页面：`EmbedReport.vue`。  
通过 query 参数渲染：

1. `type=congestion`：拥堵体检报告
2. `type=route`：路径规划报告
3. `type=nearby`：周边设施报告

页面要求：

1. 干净、简洁、平台化
2. 适合聊天窗口内嵌
3. 无广告、无扫码、无登录弹层
4. 地图画布使用高德 JS API（可按配置开关显示）

## 5. 拥堵体检模块（第一阶段实现口径）

输入示例：

1. 请分析今天主城区的拥堵情况
2. 帮我做一个城市拥堵体检
3. 早高峰为什么这么堵
4. 哪些区域最拥堵

输出结构（模板化）：

1. 基本信息：城市/区域/时段/生成时间
2. 指标卡：拥堵指数、平均车速、延误指数、异常路段数
3. 重点拥堵区域列表
4. 成因分析（规则模板）
5. 初步治理建议（规则模板）

## 6. 数据流（端到端）

1. 用户提问（聊天页）
2. Agent 识别任务类型并触发工具
3. 工具生成 `text_data + iframe_url`
4. SSE 流式返回文本与可视化片段
5. 前端在消息中展示工具标签与 iframe
6. 消息落库后，该会话历史可回看同一报告

## 7. 测试与验收

### 7.1 功能验收

1. 首页文案已切换为交通治理智能体。
2. 进入聊天页默认空白，不自动加载旧消息。
3. 提问拥堵体检类问题可触发工具并展示 iframe 报告。
4. 路线/周边同样走统一机制，不再依赖第三方原生 H5。
5. 当前会话历史刷新后可回看 iframe 报告。

### 7.2 回归项

1. 普通文本问答流式输出正常。
2. 会话新建/切换/删除不受影响。
3. 反馈按钮能力不回退。

## 8. 风险与控制

1. URL 长度风险：payload 仅保留摘要字段，列表数量限流。
2. `v-html` 风险：只渲染后端固定模板片段，不透传用户 HTML。
3. iframe 路由鉴权冲突：`/embed/report` 需避免被登录守卫误拦截。
4. 地图 SDK key 风险：统一走环境变量并按官方安全密钥方案配置。

## 9. 非目标（本阶段不做）

1. 真实交通大数据接入与复杂算法建模
2. 仿真评估算法引擎
3. 全站视觉系统重构
4. 长期报表中心与跨会话报表检索

## 10. 推荐实施顺序

1. 首页与聊天文案清理
2. 聊天默认空白会话
3. 工具统一返回结构
4. 聊天统一工具标签 + iframe 容器
5. `congestion_check` 工具
6. `/embed/report` 报告页（含高德 JS 画布）
7. 路线/周边并轨到统一机制
